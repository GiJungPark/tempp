## 공통 구현 원칙

- 레코드 길이는 문자 수가 아니라 byte 길이 기준으로 검증한다.
- 한글 필드는 CP949/MS949 기준 byte length를 맞춘다.
- CHAR/X 필드는 우측 공백 padding.
- NUMBER/9 필드는 좌측 `0` padding.
- 공란 필드는 반드시 space로 채운다.
- 합계 필드는 상세 레코드 합계와 일치해야 한다.

## 자료 출처 기준

전자자료 생성 규격의 기준은 국세청 전산매체 제출요령 원문이다.

| 업무 | 원문/보조자료 역할 |
| --- | --- |
| 원천세 | [[source/전산매체/전산매체/원천세 신고 전산매체 2faae3b52583802b8edec7c6969799e9|source 원천세 신고 전산매체 정리본]]과 원문 DOC/HWP |
| 간이지급명세서 사업소득 | [[source/전산매체/전산매체/사업소득 간이지급명세서 전산매채 2faae3b52583809f810ef113f5b80fa5|source 사업소득 간이지급명세서 정리본]]과 원문 PDF |
| 간이지급명세서 기타소득 | [[source/전산매체/전산매체/기타소득 간이지급명세서 전산매체 2faae3b5258380ee86bcf13a5948c5b4|source 기타소득 간이지급명세서 정리본]]과 원문 PDF |
| 사용자가 정리한 표 | 서비스 입력값을 각 전산매체 필드에 어떻게 매핑할지 정리한 기획/서비스 로직 보조자료 |

따라서 generator 구현 시 필드 길이, 레코드 구조, 파일명, 코드표는 원문 제출요령을 우선하고, 사용자가 정리한 표는 “우리 서비스 입력값을 어떤 필드에 넣을지” 결정하는 매핑 자료로 사용한다.
