# 공통 전자파일 생성 규칙

관련 문서: [[docs/전자자료/index|전자자료 인덱스]], [[docs/전자자료/원천세/원천세 신고 전산매체]], [[docs/전자자료/간이지급명세서/사업소득 간이지급명세서 전산매체]], [[docs/전자자료/간이지급명세서/기타소득 간이지급명세서 전산매체]]

## 목적

원천세, 사업소득 간이지급명세서, 기타소득 간이지급명세서 전자파일 generator가 공통으로 지켜야 하는 고정폭 파일 생성 규칙이다.

## 파일 형식

- 고정폭 텍스트 파일
- LINE SEQUENTIAL 형식
- 레코드마다 CR/LF값 삽입
- 레코드 순서와 각 레코드 byte 길이를 반드시 맞춘다.
- 문자열 필드는 지정 길이보다 짧으면 우측 공백 padding
- 숫자 필드는 지정 길이보다 짧으면 좌측 `0` padding
- 값이 없는 `CHAR`/`X(n)` 필드는 전체 공백
- 값이 없는 `NUMBER`/`9(n)` 필드는 전체 `0`
- 사업자등록번호와 주민등록번호는 `-`를 제거 후 수록한다.
- 전화번호는 문서별 필드 규칙을 따른다. 전산매체 원문에서 `-` 삽입을 요구하는 필드는 하이픈을 포함하고, 숫자만 요구하는 필드는 제거한다.

## 인코딩

원천세 전산매체 DOC 원문은 아래처럼 설명한다.

```text
전환방식(CONVERSION MODE): ASCII CODE
한글코드: 2Byte완성형(KSC-5601 국가표준코드)
```

실제 구현에서는 한글이 포함되는 고정폭 파일을 CP949/MS949 호환 byte length로 생성하고 검증하는 방식이 안전하다.

구현 순서:

```text
1. 필드값 문자열 준비
2. CP949/MS949 byte 기준으로 길이 검증
3. CHAR/X: CP949 byte 길이가 부족하면 space padding
4. NUMBER/9: ASCII digit만 허용하고 좌측 0 padding
5. 레코드 전체 byte length 검증
6. 줄바꿈 추가
```

주의:

- JavaScript 문자열 length는 한글 byte length와 다르다.
- 한글 필드 truncate가 필요하면 CP949 byte boundary를 깨지 않게 잘라야 한다.
- 필드 overflow는 조용히 자르지 말고 에러로 처리한다.

## 타입별 padding

| 타입 | 의미 | padding |
| --- | --- | --- |
| `CHAR`, `X(n)` | 문자/공백 포함 고정폭 | 우측 공백 |
| `NUMBER`, `9(n)` | 숫자 고정폭 | 좌측 `0` |

예:

```text
char("ABC", 5)      => "ABC  "
number(123, 5)      => "00123"
char("", 5)         => "     "
number("", 5)       => "00000"
```

## 한글/문자 작성 규칙

원문 제출요령의 공통 유의사항 기준:

- 한글 항목에는 한글로 표기한다.
- 한자, 영문자, 일본문자 등 외국문자는 소리 나는 대로 한글로 표기한다.
- 성명은 한 사람의 순수한 성명만 수록한다.
- 내국인 성명은 성과 이름 사이에 공백을 두지 않는다.
- 외국인 성명은 소리 나는 대로 한글로 풀어 쓴다.
- 법인명/상호는 좌측부터 수록하고 남는 자리는 공백으로 둔다.
- 영문 법인명은 소리 나는 대로 한글로 풀어 쓴다.
- 주소는 한글로 수록하고 좌측부터 채운 뒤 남는 자리는 공백으로 둔다.
- 아파트/건물명에 영문이 있으면 소리 나는 대로 한글로 풀어 쓴다.

## 식별번호/전화번호 작성 규칙

- 주민등록번호는 `-`를 제거하고 숫자 13자리만 수록한다.
- 주민등록번호가 없는 경우 외국인등록번호, 여권번호, 재외국민등록번호 등을 사용할 수 있으며 `-`, `/` 등 특수문자를 제거한다.
- 13문자가 안 되면 좌측부터 수록하고 우측 공백 padding한다.
- 사업자등록번호는 `-`를 제거하고 숫자 10자리만 수록한다.
- 전화번호는 해당 전산매체 원문 기준으로 하이픈 포함 여부를 결정한다.
- 하이픈 포함 필드는 지역/국/번호 사이에 `-`를 삽입한다.
- 전화번호 필드는 좌측부터 수록하고 남는 자리는 공백으로 둔다.

## 날짜/일련번호/수치 작성 규칙

- 날짜는 `YYYYMMDD`, 연월은 `YYYYMM` 형식으로 0 padding한다.
- 일련번호는 순차 부여한다.
- 일련번호는 우측 정렬 숫자이며 앞자리는 `0`으로 채운다.
- 수치항목은 유효숫자 앞을 `0`으로 채운다.
- 값이 없는 수치항목도 `0`으로 채운다.
- 음수를 허용하는 수치항목은 `-` 부호를 금액과 함께 표시한다.

예:

```text
길이 6, 양수 100  => 000100
길이 6, 음수 -100 => -00100
```

## 공통 입력 모델

```ts
interface SubmitterInput {
  taxOfficeCode?: string;
  submitDate: string; // YYYYMMDD
  submitterType: '1' | '2' | '3';
  taxAgentManagementNo?: string;
  hometaxId: string;
  businessNo: string;
  businessName: string;
  departmentName?: string;
  managerName: string;
  managerPhone: string;
}

interface PayerInput {
  taxOfficeCode: string;
  businessNo: string;
  businessName: string;
}
```

`taxOfficeCode`는 홈택스 사업장 정보에 있으면 그 값을 우선 사용한다. 없으면 `source/기준자료/국세청_세무서별_관할구역_20260408.csv`의 `세무서코드`와 `관할구역`을 기준으로 사업장 주소에서 추정한다.

## 구현 체크리스트

- 모든 레코드 byte length 검증
- 모든 필수 필드 Not Null 검증
- 숫자 필드별 음수 허용 여부 검증
- 날짜 형식 검증
- 사업자등록번호/주민등록번호 hyphen 제거
- A/B/C 레코드 집계 일치 검증
- 금액 합계 일치 검증
- 소득세/지방소득세 계산 검증
- 공란 필드는 space로 채우기

## 원문 확인 필요

- CP949 인코딩이 모든 홈택스 검증에서 요구되는지
- 각 금액 계산의 원 단위 절사/반올림 규칙
- 사업소득 업종코드별 세율 전체 코드표
- 기타소득 소득구분코드 전체 코드표
- 간이지급명세서 사업소득 지급시기 `1/2`가 월별 제출 화면에서도 항상 필요한지
- 원천세 수정/기한후 신고 시 추가 레코드 필요 여부
