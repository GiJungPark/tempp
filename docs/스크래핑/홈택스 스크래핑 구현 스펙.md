# 홈택스 스크래핑 구현 스펙

관련 문서: [[docs/스크래핑/index|스크래핑 인덱스]], [[docs/스크래핑/원천세 API 분석]], [[docs/스크래핑/간이지급명세서 API 분석]], [[docs/전자자료/index|전자자료 인덱스]]

## 목적

이 문서는 특정 프레임워크와 무관하게 홈택스 스크래핑을 다시 구현할 수 있도록, 기능별 홈택스 내부 API 호출 순서와 요청/응답 규격을 정리한다.

구현자가 준비해야 하는 공통 요소:

- HTTP client
- cookie jar
- JSON 요청/응답 처리
- `application/x-www-form-urlencoded` 요청 처리
- `multipart/form-data` 파일 업로드 처리
- Base64 인코딩/디코딩
- HMAC-SHA256
- RAONKUpload command 인코딩/디코딩

모든 기능은 같은 cookie jar를 이어서 사용해야 한다. 로그인, `permission.do`, `wqAction.do`, RAON 업로드 요청 사이에서 쿠키가 끊기면 홈택스가 정상 세션으로 보지 않는다.

## 전체 흐름

```text
1. 포털 기본 쿠키 세팅
2. 간편인증 OACX 초기화
3. 카카오 간편인증 요청
4. 사용자가 휴대폰에서 승인
5. 간편인증 결과 조회
6. pubcLogin.do로 홈택스 포털 로그인 확정
7. permission.do로 sessionMap 확보
8. 기능별 화면 permission 또는 TEHT SSO 연결
9. 기능별 wqAction.do 호출
10. 파일 업무는 RAONKUpload c01/c02/c03 후 검증 action 호출
```

## 공통 HTTP 규칙

### 기본 헤더

가능하면 브라우저와 유사하게 유지한다.

```text
User-Agent: Mozilla/5.0
Accept-Language: ko-KR,ko;q=0.9,en-US;q=0.8,en;q=0.7
Origin: https://hometax.go.kr
```

`wqAction.do` 기본 헤더:

```text
Accept: application/json
Content-Type: application/json
Origin: https://hometax.go.kr
Referer: {현재 화면 URL}
Cookie: {cookie jar}
```

```text
Accept: */*
Origin: https://hometax.go.kr
Referer: {현재 화면 URL}
Cookie: {cookie jar}
```

## 2. wqAction.do 공통 구현

홈택스 업무 API 대부분은 `wqAction.do`로 호출한다.

### URL 형식

```text
POST {baseURL}/wqAction.do?actionId={ACTION_ID}&screenId={SCREEN_ID}&popupYn={POPUP_YN}&realScreenId={REAL_SCREEN_ID}
```

예:

```text
POST https://teht.hometax.go.kr/wqAction.do?actionId=ATERNABB001A01&screenId=UTERNAAZ0Z11&popupYn=false&realScreenId=UTERNAA0Z044
```

### 헤더

```text
Accept: application/json
Content-Type: application/json
Origin: https://hometax.go.kr
Referer: {현재 화면 URL}
Cookie: {cookie jar}
```

### body 형식

홈택스는 순수 JSON body가 아니라 JSON 문자열 뒤에 NTS 검증 suffix를 붙인 문자열을 받는다.

1. payload 객체를 JSON 문자열로 만든다.
2. 숫자 타입은 문자열로 변환하는 편이 안전하다.
3. `createNtsPayload(reqData, userId)` 결과를 body로 전송한다.

```text
body = reqData + "<nts<nts>nts>" + (second + 11) + hmac + second2Digit
```

값:

```text
reqData = JSON.stringify(payload)
second = 현재 초, 0~59
second2Digit = second를 2자리 문자열로 padding
userId = sessionMap.userId
```

HMAC:

```text
algorithm = HMAC-SHA256
key = testVal[second % testVal.length]
message = reqData + userId
digest = base64
postProcess = 영숫자 이외 문자 제거
```

`testVal` 배열:

```text
0: fjaS3kdHQsdfvnm359WxzmWMV8xm5qmrcRXxolOqm4
1: qns5HuJxhT3QM8cIOSxqYw92xOpv7oMETetLjO3Zog
2: Zomr4yL5NpOcj4EfBxdDsweUxOvGWugbJ7c9xhwm
3: tOpenmvLO8XhwmY2Nxpi2eP3xcmniJj2e4xc8FamH0
4: qyVMuRUwZO93CGhkWtJFFrmEKMAg9z3FBLcKAyMxxA
5: RF413bvdLE31OL3dnmeC7r7EbMVo1oh4OrOVMMysR
6: OINbDScmre3r8ckDpIoKAyO5B6wwKulnDJkxwFBvRX
```

의사 코드:

```ts
function createNtsPayload(reqData: string, userId: string, second = nowSecond()) {
  const az = pad2(second);
  const key = testVal[second % testVal.length];
  const hmac = hmacSha256Base64(key, reqData + userId).replace(/[^0-9a-zA-Z]/g, "");
  return `${reqData}<nts<nts>nts>${second + 11}${hmac}${az}`;
}
```

### payload root 주의

WebSquare action mapping에 따라 payload root 구조가 다르다.

원천세 형식검증 `ATERNABB001A01`은 `request`로 감싸면 실패했고, 화면의 `dma_search_s1` 필드를 JSON root에 펼쳐야 했다.

```json
{
  "bsafClCd": "004",
  "itrfCd": "14",
  "orcFleNm": "..."
}
```

반대로 원천세 제출대상 조회 `ATERNABB001R06`은 아래처럼 `request` wrapper를 쓴다.

```json
{
  "request": {
    "fleSbmsCvaId": "..."
  },
  "pageInfoVO": {
    "pageSize": "10",
    "pageNum": "1",
    "totalCount": "0"
  }
}
```

최종 기준은 HAR 또는 화면 XML/JS의 submission mapping이다.


## 4. RAONKUpload 구현

전자파일 업로드는 일반 multipart 한 번으로 끝나지 않는다. RAONKUpload HTML5 프로토콜의 3단계 요청을 그대로 구현해야 한다.

### 공통 URL

```text
POST {baseURL}/fileUploadDownloadNX.do?mode=upload&uploadTypeCd={uploadTypeCd}&onlineBatch=batch
```

현재 확인값:

```text
uploadTypeCd=02
onlineBatch=batch
```

원천세:

```text
baseURL=https://teht.hometax.go.kr
referer=https://hometax.go.kr/websquare/websquare.html?w2xPath=/ui/pp/index_pp.xml&tmIdx=41&tm2lIdx=4106000000&tm3lIdx=4106010000
```

간이지급명세서 후보:

```text
baseURL=https://hometax.go.kr
referer=https://hometax.go.kr/websquare/websquare.html?w2xPath=/ui/pp/index_pp.xml&tmIdx=44&tm2lIdx=4401000000&tm3lIdx=4401100000
```

### command 구분자

```text
key-value 구분자: \f, form feed, 0x0C
필드 구분자: \v, vertical tab, 0x0B
```

필드 하나:

```text
{key}\f{value}
```

command 전체:

```text
{key}\f{value}\v{key}\f{value}\v...
```

### command 인코딩

RAON form field는 `k00`이다.

```text
k00={encodedCommand}
```

인코딩:

```text
1. command를 UTF-8 Base64 인코딩
2. 길이가 10 이상이면 아래 문자를 순서대로 삽입
   insertAt(8, "r")
   insertAt(6, "a")
   insertAt(9, "o")
   insertAt(7, "n")
   insertAt(8, "w")
   insertAt(6, "i")
   insertAt(9, "z")
3. + 문자를 %2B로 치환
```

디코딩:

```text
1. %2B를 +로 복원
2. 공백 제거
3. 앞에 _가 있으면 제거
4. 길이가 15 이상이면 [9, 6, 8, 7, 9, 6, 8] 위치 문자를 순서대로 제거
5. Base64 decode
```

### RAON 1단계: c01 시작

```http
POST {baseURL}/fileUploadDownloadNX.do?mode=upload&uploadTypeCd=02&onlineBatch=batch
Content-Type: application/x-www-form-urlencoded; charset=UTF-8
```

command:

```text
kc\fc01
k01\f0
k05\f1
k12\f{guid}
k13\f{fileSize}
k14\f{originalName}
k15\f
k16\f
k17\f{folderNameRule}
k20\f0z
k21\f
```

`folderNameRule`:

```text
/rn/UPLOAD_DIR/{YYYYMMDD}/{sessionMap.pubcUserNo}
```

응답:

```text
[OK]{encodedPlain}
```

plain:

```text
{serverPath}\v{size}\v-\v-
```

저장:

```text
serverPath
guid
folderNameRule
```

### RAON 2단계: c02 파일 조각 업로드

```http
POST {baseURL}/fileUploadDownloadNX.do?mode=upload&uploadTypeCd=02&onlineBatch=batch&raonk=urk_{random}
Content-Type: multipart/form-data
```

multipart:

```text
k00={encoded c02 command}
blob={file binary, filename=originalName, Content-Type=application/octet-stream}
```

command:

```text
kc\fc02
k01\f0
k02\f
k03\f0
k05\f1
k12\f{guid}
k19\f0
k26\f{serverPath}
```

성공 응답:

```xml
<RAONK>[OK]</RAONK>
```

### RAON 3단계: c03 완료

```http
POST {baseURL}/fileUploadDownloadNX.do?mode=upload&uploadTypeCd=02&onlineBatch=batch
Content-Type: application/x-www-form-urlencoded; charset=UTF-8
```

command:

```text
kc\fc03
k01\f0
k12\f{guid}
k14\f{originalName}
k15\f
k16\f
k17\f{folderNameRule}
k20\f0z
k21\f
```

응답:

```text
[OK]{encodedPlain}
```

plain:

```text
{originalName}:{serverRelativePath}\f
```

검증 요청에 사용할 값:

```text
uploadName = basename(serverRelativePath)
originalName = 사용자가 업로드한 파일명
serverPath = c01 응답의 전체 서버 경로
serverRelativePath = c03 응답의 /rn/... 경로
fileSize = file.size
```

검증 payload에서 보통 아래처럼 쓴다.

```json
{
  "orcFleNm": "{uploadName}",
```

## 9. 메뉴 URL / 화면 식별

국세청 안내 바로가기 후보:

```text
https://www.hometax.go.kr/websquare/websquare.html?w2xPath=/ui/pp/index_pp.xml&tmIdx=44&tm2lIdx=4401000000&tm3lIdx=4401100000
```

사용자가 공유한 검색 URL:

```text
https://hometax.go.kr/websquare/websquare.html?w2xPath=/ui/pp/index_pp.xml&menuCd=search&searchInfo2573380311
```

해석:

- `menuCd=search&searchInfo...`는 통합검색 화면의 세션 히스토리 복원용 URL이다.
- 검색 화면은 `searchInfo{random}` 키로 검색 조건을 `WebSquare.session`에 저장한다.
- 검색 결과 row 클릭 시 `MENU_ID`를 읽고 `$c.pp.goPotMenu(menuId)`를 호출한다.
- 따라서 이 URL만으로 최종 업무 화면 action을 확정할 수 없다.
- 실제 제출 화면은 캡처 기준 `UWEICAAD15`다.

## 10. 에러 처리 기준

### 홈택스 application error

예:

```json
{
  "resultMsg": {
    "msg": "입력데이터가 올바르지 않습니다.",
    "exceptType": "APPLICATION",
    "code": "ETICMZ0003",
    "result": "F"
  }
}
```

의미:

- payload root 구조가 틀렸거나
- 필수 코드값이 비었거나
- screen/action 조합이 맞지 않거나
- 세션/권한이 부족할 가능성이 높다.

### RAON failure

예:

```text
<RAONK>[FAIL]MDAxfEianzwNvorbW1hbmQ...
</RAONK>
```

디코딩하면:

```text
001|Command value is null
```

의미:

- `k00` 필드 누락
- RAON command 인코딩 오류
- `Content-Type` 또는 multipart 구성 오류
- `raonk=urk_...` query 누락 또는 command 단계 불일치

### 검증 상태 없음

`ATES...A01` 또는 `ATERN...A01` 응답에서 `trnsPrgrStat`가 없으면:

```text
1. payload root 구조 확인
2. bsafClCd / itrfCd / cvaKndCd / stmnKndCd 확인
3. uploadName, storedFileList, localfileList 확인
4. fileSizeList 확인
5. screenId / realScreenId / baseURL 확인
6. permission.do 선행 여부 확인
```
