# 원문 대조 결과

## 원문에서 확인한 핵심

### 원천세

- 파일 구조는 `LINE SEQUENTIAL`, 각 레코드마다 CR/LF를 삽입한다.
- 전환방식은 ASCII, 한글은 2바이트 완성형 KSC-5601 기준이다.
- 파일명은 `일자 + C103900 + . + 신고구분상세코드`다.
- Header 자료구분은 `21`, 서식코드는 `C103900`, 길이는 400 byte다.
- 원천징수 명세 및 납부세액 자료구분은 `23`, 길이는 150 byte다.
- 지급연월은 제출연월 - 1이어야 한다.
- 지급연월은 귀속연월보다 같거나 늦어야 한다.
- 수정/기한후 신고의 귀속연도는 현재연도 기준 5년 전까지 가능하다.
- 지급연월 기준으로 소득코드 적용시작/종료일자를 검증해야 한다.

### 사업소득 간이지급명세서

- 파일은 A/B/C 3개 레코드로 구성된다.
- 모든 레코드는 170 byte다.
- 파일명 구분자는 `SF`다.
- 파일명 예시는 사업자번호 `123-45-67890` 기준 `SF1234567.890`이다.
- C 레코드는 지급자별로 귀속연도, 귀속월, 소득자, 업종코드별 수만큼 생성한다.
- 같은 소득자에게 업종코드가 다른 사업소득을 지급하면 업종코드별로 C 레코드를 따로 만들고, 소득자 인원수도 각각 별도 인원으로 계산한다.
- A14는 B 레코드 수, B11은 C 레코드 수, B12는 C14 지급액 합계와 일치해야 한다.

### 기타소득 간이지급명세서

- 파일은 A/B/C 3개 레코드로 구성된다.
- 모든 레코드는 170 byte다.
- 파일명 구분자는 `SE`다.
- 파일명 예시는 사업자번호 `123-45-67890` 기준 `SE1234567.890`이다.
- C 레코드는 지급자별로 귀속연도, 귀속월, 소득자, 소득구분별 수만큼 생성한다.
- 원문 예시는 `76` 강연료 등과 `79` 자문료 등을 각각 별도 C 레코드로 만들고, 소득자 인원수도 2명으로 계산한다고 설명한다.
- A14는 B 레코드 수, B7은 C 레코드 수, B8은 C13 지급액 합계와 일치해야 한다.

### 사업소득 지급명세서

- 연간 사업소득 지급명세서는 A/B/C 3개 레코드로 구성된다.
- 모든 레코드는 190 byte다.
- D 레코드는 존재하지 않는다.
- 자료구분은 `24`다.
- C 레코드는 소득자별로 소득귀속연도, 지급연도, 업종구분, 세율별로 1레코드씩 금액과 건수를 합산한다.

### 기타소득 지급명세서

- 연간 기타소득 지급명세서는 A/B/C/D 4개 레코드로 구성된다.
- 모든 레코드는 300 byte다.
- 자료구분은 `23`이다.
- D 레코드는 소득구분 `64` 서화/골동품 양도소득인 경우에만 생성한다.
- 서비스 지원 기타소득 `76`, `79`에는 D 레코드를 생성하지 않는다.

## docs 반영 사항

- 간이지급명세서 파일명 규칙을 `SF/SE + 사업자번호`에서 `SF/SE + 사업자번호 앞 7자리 + . + 뒤 3자리`로 구체화했다.
- 사업소득 간이지급명세서 C 레코드 생성 단위를 `귀속연도 + 귀속월 + 소득자 + 업종코드`로 명시했다.
- 기타소득 간이지급명세서 C 레코드 생성 단위를 `귀속연도 + 귀속월 + 소득자 + 소득구분`으로 명시했다.
- 지급명세서 인덱스에 사업소득 190 byte, 기타소득 300 byte, 자료구분 `24`/`23`, 기타소득 D 레코드 조건을 반영했다.

## 남은 확인

- HWP 추출본에는 제어 문자열이 섞이므로, 연간 지급명세서 구현 전에는 원문 HWP 또는 HWP를 PDF/DOCX로 변환한 판본으로 필드표를 재검증해야 한다.
- 원천세 기획서와 변환파일 기획서의 “정기 기간 중 과거분 기한후 허용 여부” 충돌은 아직 정책 확정이 필요하다.
- 원천세 Header는 단일 귀속연월 필드인데, 기획은 행별 귀속월을 허용한다. 한 파일에 여러 귀속월을 섞을 수 있는지 홈택스 형식검증으로 확인해야 한다.
